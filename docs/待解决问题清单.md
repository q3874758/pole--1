## 二、🔴 阻断项（P0 — 不解决无法上线）

### #1 Rust → Cosmos 链交易桥接（致命）

- **问题描述**：PoLE 分两层——Rust 原型（链下数据采集、聚合、奖励计算）和 Cosmos SDK 链（链上状态机、交易处理、代币账本）。当前 Rust 原型生成的数据（批次、聚合根、奖励根）尚无法以链上交易形式提交到 Cosmos 链。
- **对应文档**：`IMPLEMENTATION_PLAN.md` 明确标注该任务「优先级高，尚未实现」
- **影响范围**：整个数据流断裂，链下采集结果无法上链，代币无法铸造/分配
- **解决思路**：
  1. 定义 Rust → Cosmos 的序列化协议（Protobuf / JSON / ABI）
  2. 节点 CLI (`pole-node`) 需集成 Cosmos 交易签名与广播能力
  3. Rust 侧生成符合链上 `MsgSubmitBatch` / `MsgCommitEpoch` 格式的交易
  4. 单元测试：Mock 链环境，验证交易构建与解析双向一致
- **预估工期**：2–4 周

---

### #2 端到端集成测试（致命）

- **问题描述**：Rust 测试（305 项）和 Cosmos 测试各自通过，但从未进行过全链路联调。从未验证「采集 → 批次承诺 → 聚合 → 奖励计算 → Challenge → Finalize → 领取」的完整闭环。
- **影响范围**：两个子系统可能因接口假定差异而无法协同工作
- **解决思路**：
  1. 搭建本地多节点测试网（2+ Rust 节点 + 1 Cosmos 验证节点）
  2. 编写自动化场景脚本：
     - 正常流程：采集 → 批次 → 聚合 → 定价 → 奖励 → 领取
     - 异常流程：挑战发起 → 挑战响应 → 挑战裁决
     - 边界场景：空批次、零权重、最高权重
  3. 验证每步链上状态一致性
- **预估工期**：1–2 周

---

### #3 第三方安全审计（致命）

- **问题描述**：`RELEASE_NOTES.md` 自身声明「按"原样"提供，建议正式运行前进行完整安全审计」。当前未进行任何外部审计。需审计的领域包括：
  - 密码学：Ed25519 签名、Merkle 树构建与验证、哈希函数使用
  - 经济模型：发行曲线、奖励分配、代币燃烧逻辑是否有算术溢出 / 舍入攻击
  - 状态机：`transitions.rs` 中的交易处理是否存在重放攻击、状态不一致
  - P2P 通信：网络层是否存在 DoS 攻击面、消息篡改风险
- **影响范围**：主网上线后发生安全事故将导致不可逆的信任损失
- **解决思路**：委托专业区块链安全审计公司进行代码审查 + 形式化验证
- **预估工期**：2–4 周（取决于审计方排期）

---

### #4 网络层安全加固（致命）

- **问题描述**：
  - `TRACEABILITY.md` 强调「Socket 模式仅在调试/测试网可用」
  - Socket 模式下通信缺少端到端加密、鉴权与会话管理
  - 生产环境缺乏完整的 TLS / Noise 协议握手、消息签名验证、防重放机制
- **影响范围**：全部链下通信可能被窃听、篡改、伪造
- **解决思路**：
  1. 为生产环境强制启用 TLS 或 Noise 协议
  2. 每条消息附带 Ed25519 签名，接收端验签
  3. 加入 nonce / timestamp 防重放
  4. 编写安全通信层专项测试（中间人攻击、重放攻击、伪造签名）
- **预估工期**：2–3 周

---

### #5 代币账本完整性验证（致命）

- **问题描述**：
  - 链下计算的奖励分配结果（`RewardRoot`）需要与链上 `MsgClaimReward` 领取严格一致
  - 目前缺少自动化工具验证「Rust 计算的奖励根」与「Cosmos 账本实际变动」的一致性
  - 账本分歧将导致玩家领取失败或超额领取
- **影响范围**：核心经济模型，账本错误将直接导致代币超发/漏发
- **解决思路**：
  1. 编写验证脚本：给定同一 epoch 数据，分别在 Rust 和 Cosmos 侧独立计算
  2. 比对奖励根哈希、个人分配数额
  3. 集成到 CI 每次 commit 自动验证
- **预估工期**：1–2 周

---

### #6 节点启动流程与创世配置标准化（致命）

- **问题描述**：
  - 缺少正式的「创世配置文件」（genesis.json）及标准化节点初始化流程
  - `node_config.rs` 配置项散乱，缺少文档，各环境（dev/testnet/mainnet）无标准配置模板
  - 新节点加入网络的操作依赖开发者手动干预
- **影响范围**：无法进行标准化部署，运维不可重复
- **解决思路**：
  1. 设计 `genesis.json` 结构：初始参数、创世节点列表、初始代币分配
  2. 实现 `pole-node init` 子命令
  3. 为 devnet / testnet / mainnet 分别提供配置模板
  4. 编写「节点加入网络」操作文档
- **预估工期**：1–2 周

### #8 CI/CD 构建流水线

- **问题描述**：无自动化编译、测试、打包流水线
- **影响范围**：手工构建易出错，发布周期不可控
- **解决思路**：
  - GitHub Actions：cargo check / build / test / clippy / fmt
  - go test 集成
  - 构建产物自动上传
  - 每个 PR 自动运行全量测试
- **预估工期**：1 周

### #9 自动更新 + 代码签名

- **问题描述**：无自动更新机制，无代码签名
- **影响范围**：安全分发困难，用户无法及时获取安全修复
- **解决思路**：
  - 签名工具集成（Windows Authenticode / GPG）
  - 版本清单 + 增量更新下载
  - 更新前校验签名
- **预估工期**：1–2 周

---

### #10 压力测试与性能基准

- **问题描述**：未对系统进行过压力测试。未知：
  - 单节点可承载的最大批次处理速率
  - 100+ 节点的区块同步延迟
  - 百万级玩家记录下的 Merkle 树构建性能
- **影响范围**：上线后可能遇到性能瓶颈导致链停滞
- **解决思路**：
  1. 模拟大量玩家（10万+）、高频率采集（万级/秒）
  2. 测量各阶段的延迟与吞吐
  3. 定位瓶颈并优化（数据库、网络、密码学）
  4. 建立性能回归测试基准
- **预估工期**：1–2 周

---

### #11 永久归档与数据保留策略

- **问题描述**：挑战窗口之后的数据「不保证永久保留」。但对于争议追溯，可能需要在数月甚至数年后重放历史数据
- **影响范围**：长期争议无法举证，信任受损
- **解决思路**：
  1. 明确数据保留策略：挑战窗口结束后至少保留 N 个 epoch
  2. 可选：IPFS / Arweave 归档历史批次和聚合数据
  3. 归档索引可查，支持选择性重放
- **预估工期**：1–2 周

---

### #12 运维监控面板

- **问题描述**：无节点健康监控、无区块生产监控、无告警
- **影响范围**：故障排查依赖开发者现场介入
- **解决思路**：
  - Prometheus + Grafana 集成
  - 关键指标：区块高度、奖励池余额、活跃节点数、挑战胜率
  - 告警规则：出块停滞、节点掉线、异常挑战激增
- **预估工期**：1–2 周

---

### #13 灾难恢复方案

- **问题描述**：无文档说明「如果主网出问题怎么办」：
  - 链回滚策略
  - 节点数据恢复
  - 紧急参数更新（治理快速通道）
- **影响范围**：线上事故无预案，响应混乱
- **解决思路**：编写灾难恢复手册，覆盖 5+ 典型故障场景，每场景有明确的恢复步骤和责任人
- **预估工期**：1 周
