#### Step-by-step 实现

**Step 1: 地址格式统一 (1 天)**

```rust
// src/cosmos/address.rs

use bech32::{encode, decode, FromBase32, ToBase32, Variant};

const BECH32_PREFIX: &str = "cosmos";

/// Rust hex NodeId → Cosmos bech32 address
pub fn hex_to_bech32(hex: &str) -> Result<String> {
    let bytes = hex::decode(hex)?;
    encode(BECH32_PREFIX, bytes.to_base32(), Variant::Bech32)
}

/// Cosmos bech32 → Rust hex NodeId
pub fn bech32_to_hex(addr: &str) -> Result<String> {
    let (_, data, _) = decode(addr)?;
    let bytes = Vec::<u8>::from_base32(&data)?;
    Ok(hex::encode(bytes))
}
```

需要在 `Cargo.toml` 添加 `bech32` 依赖。

> **注意**：现有 Rust 代码中 `NodeId` 是 `[u8; 32]`（Hash32），链上用 Cosmos SDK 地址（20 字节）。需在桥接层做适配：`NodeId` 取前 20 字节转为 bech32 作为 Cosmos 地址，或新增 `CosmosAddress` 类型。

**Step 3: Protobuf 交易构造 (3 天)**

关键问题：Rust 端需要按 Cosmos SDK 的 protobuf 格式构造交易。

Cosmos SDK 交易结构：

```
Tx {
    body: TxBody {
        messages: [Any],      // 含 MsgSubmitBatch 等
        memo: "",
        timeout_height: 0,
        extension_options: [],
        non_critical_extension_options: [],
    },
    auth_info: AuthInfo {
        signer_infos: [SignerInfo],
        fee: Fee { amount, gas_limit, payer, granter },
    },
    signatures: [bytes],
}
```

实现方案 A（推荐）：**手动构造 JSON + 序列化**

Cosmos SDK 的 REST API 支持 JSON 编码的交易。可以：
1. 用 Rust 构造交易的 JSON 表示
2. 通过 `POST /cosmos/tx/v1beta1/txs` 广播

```rust
impl TxBuilder {
    pub fn build_submit_batch_tx(
        collector_addr: &str,
        batch: &BatchCommit,
        account_number: u64,
        sequence: u64,
        chain_id: &str,
    ) -> Result<Vec<u8>> {
        // 1. 构造 MsgSubmitBatch 的 Any 类型
        let msg = serde_json::json!({
            "@type": "/pole.chain.pole.v1.MsgSubmitBatch",
            "collector": collector_addr,
            "batch_commit": batch_commit_to_cosmos_json(batch),
        });

        // 2. 构造 TxBody
        let body = TxBody {
            messages: vec![msg],
            memo: "".to_string(),
            timeout_height: 0,
            // ...
        };

        // 3. 构造 AuthInfo (含 fee)
        let auth_info = AuthInfo {
            signer_infos: vec![SignerInfo {
                public_key: Some(public_key_to_any(pubkey)),
                mode_info: ModeInfo::single(cosmos_sdk::tx::signing::SignMode::Direct),
                sequence,
            }],
            fee: Fee {
                amount: vec![Coin { denom: "upole".into(), amount: "2000".into() }],
                gas_limit: 200_000,
            },
        };

        // 4. 构造 SignDoc + 签名
        let sign_doc = SignDoc { body_bytes, auth_info_bytes, chain_id, account_number };
        let sig = keypair.sign(&sign_doc.to_bytes());

        // 5. 组装 TxRaw
        TxRaw { body_bytes, auth_info_bytes, signatures: vec![sig] }.encode()
    }
}
```

实现方案 B（备选）：**调用 Cosmos SDK 的 sign-mode-textual**

如果手动序列化复杂，可以在 Rust 中调用 `poled` 二进制来实现签名（通过 `std::process::Command`），但会增加部署依赖。

**Step 5: 错误处理与重试 (2 天)**

```rust
pub struct BroadcastOptions {
    pub max_retries: u32,
    pub retry_delay_secs: u64,
    pub timeout_secs: u64,
}

impl TendermintRpc {
    pub async fn broadcast_with_retry(&self, tx: &[u8], opts: &BroadcastOptions) -> Result<TxHash> {
        for attempt in 1..=opts.max_retries {
            match self.broadcast_tx_sync(tx).await {
                Ok(resp) if resp.code == 0 => return Ok(resp.txhash),
                Ok(resp) => warn!("tx failed with code {}: {}", resp.code, resp.log),
                Err(e) => warn!("broadcast attempt {}/{}: {}", attempt, opts.max_retries, e),
            }
            sleep(Duration::from_secs(opts.retry_delay_secs)).await;
        }
        Err(BridgeError::BroadcastFailed)
    }
}
```

### 2.5 最小可验证集成测试（优先实现）

作为 MVP，先实现 **场景 1+2+4+5**（约占全流程的 80%）：

```rust
#[tokio::test]
async fn test_full_epoch_lifecycle() {
    let h = IntegrationHarness::new().await;

    // 1. 注册节点
    let node_id = h.register_node("player").await;
    assert!(h.is_node_active(&node_id).await);

    // 2. 提交 12 个 tick 的批次数据（= 1 小时区块）
    for tick in 0..12 {
        let txhash = h.submit_batch(&node_id, tick).await.unwrap();
        assert!(!txhash.is_empty());
    }

    // 3. 提交 epoch commit
    let epoch = h.commit_epoch(&node_id, 1).await.unwrap();
    assert_eq!(epoch.epoch_id, 1);

    // 4. 等待挑战窗口结束
    h.wait_blocks(20).await;

    // 5. 终结 epoch
    h.finalize_epoch(1).await.unwrap();

    // 6. 领取奖励
    let balance_before = h.get_balance(&node_id, "upole").await;
    h.claim_reward(&node_id, 1).await.unwrap();
    let balance_after = h.get_balance(&node_id, "upole").await;
    assert!(balance_after > balance_before, "reward should increase balance");

    // 7. 验证奖励金额与 Rust 本地计算结果一致
    let local_reward = h.get_local_reward(&node_id, 1).await;
    let chain_reward = balance_after - balance_before;
    assert!(
        (local_reward as i128 - chain_reward as i128).abs() <= 1,
        "local={}, chain={}", local_reward, chain_reward
    );
}
```

### 4.2 安全加固方案

#### Phase 1: 补丁加固（短期，1 周）

给 Socket P2P 加最小安全层：

```rust
// 使用 Noise 协议框架做握手+加密
use snow::{Builder, HandshakeState};

pub struct SecureSocketP2p {
    inner: TcpStream,
    noise: NoiseSession,
    peer_pubkey: [u8; 32],
}

impl SecureSocketP2p {
    pub fn connect(addr: &str, my_keypair: &KeyPair, peer_pubkey: &[u8; 32]) -> Result<Self> {
        let mut stream = TcpStream::connect(addr)?;

        // Noise XX 握手
        let mut noise = Builder::new("Noise_XX_25519_ChaChaPoly_BLAKE2s".parse()?)
            .local_private_key(&my_keypair.secret)
            .remote_public_key(peer_pubkey)
            .build_initiator()?;

        // 握手消息交换
        let mut buf = [0u8; 1024];
        let len = noise.write_message(&[], &mut buf)?;
        stream.write_all(&buf[..len])?;

        let n = stream.read(&mut buf)?;
        noise.read_message(&buf[..n], &mut [])?;

        // 握手完成后所有通信加密
        Ok(Self {
            inner: stream,
            noise: noise.into_transport_mode()?,
            peer_pubkey: *peer_pubkey,
        })
    }

    pub fn send(&mut self, data: &[u8]) -> Result<()> {
        let mut buf = vec![0u8; data.len() + 16]; // tag overhead
        let len = self.noise.write_message(data, &mut buf)?;
        // 前缀长度 + 加密数据
        self.inner.write_all(&(len as u32).to_le_bytes())?;
        self.inner.write_all(&buf[..len])?;
        Ok(())
    }
}
```

#### Phase 2: 切换到 LibP2P（中期，2-3 周）

```rust
// 使用 rust-libp2p 的完整安全栈
use libp2p::{
    noise, yamux, tcp,
    identity::Keypair as Libp2pKeypair,
    swarm::Swarm,
};

pub fn build_secure_libp2p_swarm(
    keypair: &ed25519_dalek::SigningKey,
) -> Swarm<MyBehaviour> {
    let local_key = Libp2pKeypair::from(
        libp2p::identity::ed25519::Keypair::from(
            libp2p::identity::ed25519::SecretKey::try_from_bytes(keypair.to_bytes()).unwrap()
        )
    );

    let noise = noise::Config::new(&local_key).unwrap();
    let transport = tcp::tokio::Transport::default()
        .upgrade(libp2p::core::upgrade::Version::V1)
        .authenticate(noise)
        .multiplex(yamux::Config::default());

    Swarm::new(transport, MyBehaviour::new(), local_key.public().to_peer_id())
}
```

## P0-5 账本对齐方案

### 5.1 现状诊断

Rust 原型和 Cosmos 链各自维护独立但相关的状态账本：

| 数据项 | Rust 端 | Cosmos 链端 | 对齐方式 |
|--------|---------|-------------|---------|
| epoch_id | 本地计数器 | 链上 `epochs` keeper | 需确保相同 epoch 指向相同时间窗口 |
| block_height | 本地推算 | CometBFT 实际高度 | 桥接时用链上高度 |
| 节点 ID/地址 | hex (64 char) | bech32 (如 `cosmos1...`) | ✅ 已有转换函数可对齐 |
| 奖励金额 | 本地计算 | 链上 keeper 计算 | 公式相同，需验证 Rounding |
| 批次 Merkle 根 | 本地计算 | 链上存储 | 需完全一致 |
| 时间戳 | UnixMillis | CometBFT block time | ±1 小时窗口内对齐 |

### 5.2 对齐策略

#### 策略 A：以链为准（推荐）

Rust 作为「预言机」角色，将计算结果提交给链。链具有最终裁决权。

```rust
// Rust 提交前先查询链上状态
let chain_height = rpc.latest_height().await?;
let chain_epoch = rpc.query_current_epoch().await?;

// 确保本地 epoch 与链上同步
assert_eq!(
    local_epoch, chain_epoch,
    "Local epoch {} != chain epoch {}",
    local_epoch, chain_epoch
);

// 用链上高度作为 submission_height
let batch = BatchCommit {
    epoch_id: chain_epoch,
    submitted_at_height: chain_height,
    // ...
};
```

#### 策略 B：离线文件 + 手动校对（过渡方案）

在桥接完全自动化之前，Rust 生成 JSON 文件，由运维人员检查后手动提交。

```bash
# 1. Rust 生成 JSON
pole-node export batch --epoch 42 --output batch-42.json

# 2. 运维检查（对照链上数据）
diff <(cat batch-42.json | jq '.epoch_id') <(poled query pole current-epoch)

# 3. 手动广播
poled tx pole submit-batch $(cat batch-42.json | jq -c) --from validator
```

### 5.3 关键对齐点详解

#### A. Rounding 一致性

```rust
// Rust 端：整数除法
let reward = pool * weight / total_weight;  // 截断

// Go 端 (sdk.Dec)：
reward := pool.Mul(weight).Quo(total_weight).TruncateInt()

// 验证脚本
fn verify_rounding() {
    for pool in 1..1000 {
        for weight in 1..pool {
            for total in weight..1000 {
                let rust = pool * weight / total;
                let go = pool * weight / total;  // 需要实际调用 Go 代码
                assert_eq!(rust, go, "Rounding mismatch");
            }
        }
    }
}
```

#### B. Merkle 树一致性

Rust 和 Go 端必须使用相同的 Merkle 树构造逻辑（相同哈希函数、相同叶节点序）。

```rust
// 验证脚本
fn verify_merkle_consistency() {
    let leaves = vec![b"a", b"b", b"c"];
    let rust_root = build_merkle_root(&leaves);  // SHA-256

    // 在 Go 侧用相同数据构造
    let go_root = query_go_merkle("a", "b", "c");

    assert_eq!(rust_root, go_root);
}
```

#### C. 时间窗口对齐

epoch_id 映射到 UTC 小时：

```rust
fn epoch_id_for_timestamp(millis: u64) -> u64 {
    // epoch_id = 自创世时间起的小时数
    const GENESIS_EPOCH_MS: u64 = 1_700_000_000_000;  // 需配置
    (millis - GENESIS_EPOCH_MS) / 3_600_000
}
```

Rust 和 Go 端必须使用相同的 `GENESIS_EPOCH_MS`。
